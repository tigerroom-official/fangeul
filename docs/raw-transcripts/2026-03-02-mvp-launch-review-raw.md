# Expert Panel - 범용 전문가 토론 엔진

어떤 주제든 전문가 패널이 **실시간으로 토론**하여 합의와 액션 아이템을 도출합니다.

## 핵심 원칙

**사용자가 토론 과정을 실시간으로 볼 수 있어야 합니다.**

- 최종 요약만 보여주지 않음
- 토픽별로 A조/B조가 공격-반론-재반론 하는 과정을 출력
- 사용자가 토론 중 개입하거나 방향을 조정할 수 있음

### Step 2: 전문가 페르소나 자동 제안 (프리셋이 없을 때만)

프리셋이 매칭되지 않으면, 주제에 맞는 전문가 패널을 자동 제안합니다.

**제안 원칙:**
- 3~6명 제안 (주제 복잡도에 따라 조절)
- 각 전문가: 이름, 역할, 우선 가치, 성격/말버릇
- **진정으로 의견이 갈리는** 전문가 조합 (모두 같은 의견이면 토론이 안 됨)
- 최소 1명은 **회의론자** (건설적 비판 담당)
- 이해관계가 다른 전문가 포함 (개발자 vs 운영, 성능 vs 안정성 등)

**출력 형식:**

```markdown
## 전문가 패널 제안

주제 "{topic}"에 대해 다음 전문가 패널을 제안합니다.

| # | 이름 | 역할 | 우선 가치 | 성격/말버릇 |
|---|------|------|----------|-------------|
| 1 | {이름} | {역할} | {가치1} > {가치2} | "{말버릇}" |
| 2 | ... | ... | ... | ... |

이 구성으로 진행할까요?
```

**AskUserQuestion**으로 사용자 승인 후 진행.

### Step 4: 토픽별 실시간 토론

**각 토픽마다 다음 4단계를 반복합니다.**

Task agent를 병렬로 실행하되, 결과를 취합하여 토론 형식으로 출력합니다.

#### 4.1 A조 선공 (병렬 실행)

A조에 배정된 전문가들이 **지지/찬성** 관점에서 분석합니다.

**Agent 프롬프트 템플릿:**

```
당신은 **{전문가명}**, {역할}입니다.

## 성격
{성격 및 말버릇}

## 우선 가치
{우선 가치}

## 논제
{topic}

## 배경
{context}

## 당신의 역할
이 토픽에 대해 **지지/찬성** 관점에서 분석해주세요.
긍정적 측면, 장점, 실현 가능성을 중심으로 논증하되,
당신의 전문 분야 관점에서 구체적인 근거를 제시하세요.

## 출력 형식
### 핵심 주장
[1-2문장으로 핵심 주장]

### 근거
1. [구체적 근거 1]
2. [구체적 근거 2]
3. [구체적 근거 3]

### 예상 반론에 대한 선제 대응
[B조가 제기할 수 있는 반론에 미리 대응]
```

**출력 형식:**

```markdown
---

## Topic {N}: {토픽명}

### A조 선공

**{전문가1} ({역할}):**

> "{특유의 말버릇}"

{분석 내용}
- {근거 1}
- {근거 2}

**{전문가2} ({역할}):**

> "{특유의 말버릇}"

{분석 내용}
- {근거 1}
- {근거 2}

---
```

#### 4.2 B조 반론 (병렬 실행)

B조 전문가들이 A조 주장에 대해 **반론**합니다.

**Agent 프롬프트에 추가:**

```
## A조 주장 요약
{a_team_summary}

## 당신의 역할
A조의 주장에 대해 **반론**을 제기해주세요.
우려 사항, 위험 요소, 실현 가능성 문제를 중심으로 논증하세요.

## 출력 형식
### A조 주장 중 동의하는 부분
[부분적 동의 - 완전 반대는 비생산적]

### 핵심 반론
[1-2문장]

### 근거
1. [반론 근거 1]
2. [반론 근거 2]

### 대안 제시
[단순 반대가 아닌 건설적 대안]
```

**출력 형식:**

```markdown
### B조 반론

**{전문가3} ({역할}):**

> "{특유의 말버릇}"

{반론 내용}
- A조 주장 중 동의: {동의 부분}
- 하지만: {핵심 반론}
- 대안: {대안 제시}

**{전문가4} ({역할}):**

> "{특유의 말버릇}"

{반론 내용}
- 실제 사례: {구체적 사례}
- 우려 사항: {우려 내용}

---
```

#### 4.3 A조 재반론 (병렬 실행)

A조가 B조 반론에 대해 **재반론**합니다.

**출력 형식:**

```markdown

- 결론: {보완된 주장}

**{전문가2}:**

> "{특유의 말버릇}"

{재반론 내용}

---
```

### 1. 30%→43% — 카드 스와이프 맥락 수정

맞습니다. 카드 탐색은 **좌우 PageView 스와이프**라서 "세로로 카드 1.3개 보인다"는 잘못된 논거였습니다. 실제로 43%로 높이면:
- 카드 자체가 **더 여유있게** 표시 (한국어 + 로마자 + 버튼이 덜 빡빡)
- 팩 필터 칩이 잘리지 않고 **완전히 노출**
- 드래그 핸들 + 탭바 + 필터 + 카드가 모두 **숨 쉴 공간** 확보

"더 많은 카드"가 아니라 "현재 카드를 더 편하게 보여주는" 개선입니다. 여전히 유효하지만 논거 수정이 필요합니다.

### 2. 마지막 사용 팩 — 미구현 확인

확인 결과 **미구현**입니다:
- `CompactPhraseFilterNotifier.build()` → 항상 `CompactPhraseFilter.favorites()` 반환
- SharedPreferences 저장/복원 로직 없음
- 버블 열 때마다 즐겨찾기 탭으로 초기화

착각하신 것 같습니다. 구현이 필요한 항목 맞습니다.

### 3. K-pop 캘린더 — Claude Code CLI 자동화

훨씬 현실적인 접근입니다. 정리하면:

```
[공개 K-pop 캘린더/위키/스케줄 사이트]
         ↓ 크롤링 or 캘린더 MCP
[Claude Code CLI (Max 구독)]
         ↓ 프롬프팅 → 일관된 포맷 변환
[assets/calendar.json]
         ↓ 앱 빌드에 포함
[날짜 매칭 → 상황 태그 서페이싱]
```

수동 입력이 아니라 **반자동 파이프라인**:
- 크롤링/MCP로 원본 수집
- Claude Code로 정제 + JSON 생성
- 주기적 갱신 (월 1회 or 시즌별)
- 앱 업데이트 or Remote Config로 배포

Codex가 우려한 "솔로 개발자 수동 입력 비현실적" 문제를 해결하면서도, 풀 자동화의 정확도 리스크를 피하는 중간 지점입니다.
