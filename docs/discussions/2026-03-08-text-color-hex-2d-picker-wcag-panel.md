## A조 선공

---

**유리 (UX 디자이너):**

"유저 입장에서 3초 안에 이해 안 되면 실패예요. 지금 글자색 선택 UI를 보세요. `_TextColorSelector`에 프리셋 6개 원형 버튼 + '자동' + '+' 버튼. '+' 누르면 `TextColorPickerDialog`가 뜨는데, 거기서 할 수 있는 게 hue 슬라이더 하나, tone 슬라이더 하나. chroma는 20으로 고정이에요.

BTS 팬이 `#FFD700` 골드를 글자색으로 쓰고 싶다? 이 UI로는 **절대 불가능**해요. HCT hue 슬라이더를 아무리 돌려도 chroma 20 고정이면 파스텔톤밖에 안 나와요. `#FFD700`은 chroma가 대략 85 정도 되는 고채도 색상인데, 현재 피커에선 도달할 수 없는 영역이에요.

제 제안은 세 가지예요:

**첫째, hex 입력 필드 추가.** `TextColorPickerDialog` 상단에 `#` prefix + 6자리 TextField. 팬들은 자기 최애 그룹 공식 컬러 hex를 이미 외우고 다녀요. 트위터 프로필, 팬카페 배경, 다 hex로 공유하죠. `#FFD700` 붙여넣기 한 방이면 끝나야 해요. 실시간 파싱으로 TextField 옆에 색상 프리뷰 원을 보여주고, 유효하지 않은 hex면 원을 회색으로.

**둘째, 2D 피커 재사용.** 배경색용 `HctColorPicker`가 이미 있어요. X=chroma, Y=tone, 하단 hue 바. 이걸 글자색 선택에도 그대로 쓰면 돼요. 별도 경량 피커를 만들 이유가 없어요. 통일된 경험이 학습 비용을 줄여요. '아, 배경색 고를 때 쓰던 그 사각 피커네' — 즉시 이해.

**셋째, WCAG 추천색 통합.** 지금 `suggestTextColors()`가 이미 있잖아요. 배경색 기반으로 보색, 유사색, 무채색에서 4.5:1 만족하는 후보를 반환하는. 이걸 `TextColorPickerDialog` 안에 '추천 색상' 행으로 넣으면, 유저가 자유 선택과 안전 선택을 한 화면에서 비교할 수 있어요."

---

**하은 (접근성 컨설턴트):**

"예쁜데 못 읽으면 예쁜 게 아니에요. 유리 님 제안 전부 동의하는데, WCAG 경고 부분을 더 강하게 가져가고 싶어요.

지금 구현을 보면 — `theme_picker_sheet.dart` 289행 근처에 `contrastRatio(textColor, bgColor)` < 4.5일 때 Chip으로 경고를 보여주고 있어요. `TextColorPickerDialog`에도 114행에 대비율을 실시간 표시하고 있고요. **하지만 이 두 경고가 서로 다른 맥락에서 따로 놀아요.**

시트에서의 경고는 '이미 선택한 후'에 보이고, 다이얼로그 안에서의 경고는 '선택 중'에 보여요. 문제는 유저가 다이얼로그에서 낮은 대비의 색을 고르고 '완료'를 눌렀을 때, 시트로 돌아와서야 또 경고를 보게 된다는 거예요. **이중 경고인데 정작 막아주지는 않는** 어정쩡한 상태.

제 제안:

**WCAG 경고 체계를 3단계로 분류하자:**

| 대비율 | 레벨 | UI 피드백 | 차단 여부 |
|--------|------|-----------|-----------|
| >= 4.5:1 | AA 충족 | 초록 체크 아이콘 + 비율 숫자 | 통과 |
| 3.0 ~ 4.49:1 | AA 미달, 대문자 OK | 주황 경고 아이콘 + '가독성 주의' 텍스트 | 경고만 |
| < 3.0:1 | 거의 읽을 수 없음 | 빨강 경고 아이콘 + '텍스트 보이지 않을 수 있음' + 프리뷰 강조 | 경고만, **하지만 확인 단계 추가** |

3.0 미만은 사실상 실제 기기에서 텍스트가 안 보여요. 이건 '개성 표현'이 아니라 '기능 장애'예요. 이전 패널에서 '차단 금지'로 합의했으니 적용을 막지는 않되, **`< 3.0`에서 '완료' 누르면 '이 색상은 텍스트가 보이지 않을 수 있습니다. 적용하시겠습니까?' 확인 다이얼로그를 한 번 띄우자**는 거예요. 이건 차단이 아니라 '확인'이에요. 유저 자율성을 존중하면서도 무의식적 실수를 방지해요.

그리고 hex 입력 관련으로 — hex로 `#000000`을 입력하고 배경도 검은색이면? 대비율 1.0:1이에요. hex 입력에도 실시간 WCAG 피드백이 필수예요. TextField 아래에 대비율 바를 깔아서, 입력하는 순간순간 색이 바뀌면서 대비율도 업데이트되게."

## B조 반론

---

**준서 (Flutter 개발자):**

"동작하는 코드가 아름다운 코드를 이깁니다. 두 분 제안을 하나씩 짚어볼게요.

**hex 입력 — 구현은 간단하지만 엣지케이스가 많아요.**

TextField에 `#FFD700` 입력하면 파싱해서 색상 프리뷰 보여주는 거 자체는 쉬워요. `int.tryParse(hex, radix: 16)` 한 줄이면 되니까. 근데 실제 유저 행동을 생각해보세요:

- `#` 포함/미포함: `FFD700` vs `#FFD700`
- 3자리 축약: `#FD0` -> `#FFDD00`?
- 8자리 ARGB: `#80FFD700` (반투명)
- 소문자/대문자: `ffd700` vs `FFD700`
- 중간에 공백이나 특수문자 붙여넣기: `#FFD 700`
- 한글 자판 상태에서 입력 시도

이거 다 핸들링해야 해요. `InputFormatter`로 영숫자만 허용하고, 길이 6으로 제한하고, `#` prefix는 장식용으로 TextField 밖에 두고... 복잡도가 올라가요.

**2D 피커 재사용 — 여기가 제일 우려되는 부분이에요.**

`HctColorPicker`를 보세요. 128x96 해상도로 HCT 2D 이미지를 픽셀 버퍼로 생성해요. `_generateHctImage()`에서 `Hct.from()` 호출을 128*96 = 12,288번 해요. hue 변경할 때마다 재생성이고요. 배경색 피커에서는 이게 한 번만 열리니까 괜찮은데, **글자색 피커에서도 이걸 쓰면 같은 시트 안에서 2D 이미지가 두 개 돌아가는 거예요.**

물론 hue 변경 시에만 재생성되고, 이미지 캐시도 있어요. 하지만 `TextColorPickerDialog`는 `AlertDialog`로 열리는데, 그 안에 160px 높이의 2D 영역을 넣으면 다이얼로그 높이가 폭발해요. `AlertDialog` `content`에 Column으로 프리뷰 + WCAG 표시 + 2D 피커 + hue 바 + hex 입력 + 추천색... 스크롤이 필요하게 되는데, AlertDialog 안에서의 스크롤은 UX가 좋지 않아요.

**차라리 TextColorPickerDialog를 AlertDialog에서 바텀시트로 바꾸거나**, 아니면 2D 피커의 경량 버전을 만들어야 해요. 해상도를 64x48로 줄이든, 피커 높이를 120px로 줄이든.

**WCAG 3단계 + 확인 다이얼로그 — 이건 동의하기 어려워요.**

하은 님, `< 3.0`에서 확인 다이얼로그를 띄우자는 건 사실상 차단이에요. 유저가 자유 피커에서 색을 고르고 '완료'를 눌렀는데 또 다이얼로그? 이건 '진짜 할 거야?'라고 묻는 거잖아요. 이전 패널에서 '경고 전용, 차단 아님'으로 합의했어요. 확인 다이얼로그는 그 합의의 경계를 넘어요.

유저가 일부러 낮은 대비를 원하는 케이스가 있어요. 예를 들어 글자색을 배경과 거의 같은 톤으로 맞춰서 '은은한 워터마크' 느낌을 만드는 팬도 있거든요. 스크린샷 공유할 때 배경색만 보이게 하는 용도로. 그런 유저에게 매번 확인 다이얼로그는 짜증 유발이에요."

---

**소영 (M3 디자인 엔지니어):**

"Material 스펙을 벗어나면 유지보수가 눈덩이처럼 불어나요. 몇 가지 구조적 우려를 말씀드릴게요.

**첫째, textColorOverride의 전파 범위 문제.**

`CustomSchemeBuilder.build()`를 보세요. `textColorOverride`가 null이 아니면 `onSurface`, `onSurfaceVariant`, `onPrimary`, `onPrimaryContainer`, `onSecondary`, `onSecondaryContainer`, `onTertiary`, `onTertiaryContainer`, `onError`, `onErrorContainer` — 총 10개 슬롯을 전부 같은 색으로 덮어씌워요. `onSurfaceVariant`만 alpha 0.78로 살짝 낮추고요.

문제는 이거예요: **유저가 hex로 `#FFD700` (골드)를 글자색으로 설정하면, 에러 메시지의 텍스트도 골드, SnackBar의 텍스트도 골드, Dialog의 버튼 텍스트도 골드가 돼요.** 에러 상황에서 골드색 텍스트? 시맨틱하게 완전히 깨져요. '에러인데 왜 이렇게 예쁜 색이지?'

Material 3에서 `onError`는 에러 컨테이너 위의 텍스트 색상이에요. 이걸 유저 선택색으로 덮으면 에러 메시지의 가독성과 의미 전달이 동시에 파괴돼요. `onPrimary`도 마찬가지 — 버튼 위의 텍스트인데, `primaryContainer` 배경과 유저 선택 글자색의 대비를 보장할 수 없어요.

현재 WCAG 검사는 `surface` 대비만 보고 있어요. 하지만 유저 글자색이 적용되는 곳은 `primaryContainer`, `secondaryContainer`, `errorContainer`, `surface` 등 **배경이 전부 달라요.** 한 배경에서 4.5:1 통과해도 다른 배경에서는 2.0:1일 수 있어요.

**둘째, hex 입력과 2D 피커의 상태 동기화.**

hex 입력으로 색을 정하면 2D 피커의 마커 위치도 업데이트되어야 해요. 그런데 hex 입력은 sRGB 공간이고, 2D 피커는 HCT 공간이에요. `#FFD700` -> HCT 변환 -> 피커 위의 마커 위치... 이건 1:1 대응이 안 될 수 있어요. HCT에서 chroma가 gamut 경계에서 클램핑되면, 피커의 마커 위치에서 나오는 색이 입력한 hex와 미세하게 달라져요. 유저는 '내가 입력한 색이랑 다른데?' 하게 돼요.

**셋째, 글자색용 2D 피커의 필요성 자체에 의문.**

배경색은 앱 전체의 '분위기'를 결정하니까 미세 조정이 중요해요. 하지만 글자색은 **가독성이 핵심**이에요. 대부분의 유저가 원하는 건 '정확히 이 hex 값' 아니면 '추천 중에서 고르기'예요. HCT chroma x tone 2D 평면에서 글자색을 탐색할 유저가 과연 몇 %일까요?

hex 입력 + 추천색 + 기존 hue/tone 슬라이더(chroma 범위만 확장)면 충분하지 않을까요? 2D 피커까지 넣으면 다이얼로그가 너무 무거워져요."

## A조 재반론

---

**유리 (UX 디자이너):**

"준서 님, 소영 님 우려 잘 들었어요. 하나씩 응답할게요.

**hex 엣지케이스 — 맞아요, 하지만 해결된 패턴이에요.** `#` prefix는 TextField 밖에 `Text('#')` 위젯으로 두고, TextField는 6자리 영숫자만 받으면 돼요. `FilteringTextInputFormatter.allow(RegExp(r'[0-9a-fA-F]'))` + `LengthLimitingTextInputFormatter(6)`. 3자리 축약은 지원 안 하면 돼요 — 팬들이 공유하는 hex는 항상 6자리예요. 8자리 ARGB? 알파 채널은 글자색에서 의미 없으니 앞 2자리 잘라서 6자리만 쓰면 되고, 그것도 MVP에서는 6자리 정확 매칭만 하면 충분해요. **과도 설계를 피하자는 준서 님 철학에 오히려 부합해요.**

**2D 피커 vs AlertDialog 크기 — 이건 제가 양보할 수 있어요.** 소영 님 말씀이 맞아요. 글자색에 2D chroma x tone 탐색이 필요한 유저는 소수예요. 대부분은 'hex 붙여넣기' 아니면 '추천에서 고르기'예요.

그래서 **수정 제안**: `TextColorPickerDialog`를 2D 피커 없이, **hex 입력 + hue/tone 슬라이더(chroma 범위 확장) + 추천색 행**으로 구성하자는 거예요. 현재 chroma 20 고정을 **hue 슬라이더 + chroma 슬라이더 + tone 슬라이더 3개**로 바꾸는 게 아니라, **chroma를 configurable하게 만들되 기본값 60, 범위 10~100**으로 하고 토글로 숨겨두자는 거예요. 일반 유저는 hue+tone만 쓰고, '더 진하게/연하게' 필요한 유저만 chroma 조절. 이러면 `#FFD700` 같은 고채도 색도 도달 가능하면서, UI 복잡도는 관리 가능해요.

그런데 실은... **hex 입력이 있으면 슬라이더를 쓸 필요 자체가 줄어요.** 정확한 값을 아는 유저는 hex를 입력하고, 탐색하고 싶은 유저는 슬라이더를 쓰고. 두 경로가 공존하면 돼요."

**하은 (접근성 컨설턴트):**

"준서 님, 확인 다이얼로그 = 차단이라는 의견... 일리가 있어요. 하지만 구분하고 싶은 게 있어요.

**3.0:1 미만은 '선택'이 아니라 '실수'일 확률이 압도적으로 높아요.** W3C WCAG 2.1에서 3.0:1은 'Large Text(18pt+) 최소 기준'이에요. 14~16px 본문 텍스트에서 3.0:1 미만이면 **시력이 좋은 사람도** 밝은 환경에서 읽기 어려워요. 준서 님이 말씀하신 '은은한 워터마크' 용도 — 그건 극소수 파워유저의 의도적 사용이고, 대다수 유저가 3.0 미만을 적용하는 건 '이 색 예쁘다' 하고 골랐는데 실제 기기에서 안 보이는 결과를 초래해요.

**하지만**, 이전 합의를 존중해서 양보할게요. 확인 다이얼로그 대신 **inline 경고 강화**로 가겠습니다:

- `< 3.0:1`: 프리뷰 영역에 **반투명 오버레이 + '텍스트가 보이지 않을 수 있습니다' 텍스트**를 프리뷰 위에 겹쳐서 표시. 확인 다이얼로그 없이 '완료' 가능하되, **프리뷰 자체가 경고**가 되는 거예요. 유저가 프리뷰를 보면서 '아, 진짜 안 보이네' 스스로 인지하게.
- `3.0 ~ 4.49:1`: 현재처럼 주황 아이콘 + 비율 표시.
- `>= 4.5:1`: 초록 체크.

이러면 차단도 아니고, 확인 다이얼로그도 아니고, 유저가 시각적으로 문제를 인지하게 만드는 거예요.

소영 님의 `onError` 시맨틱 파괴 문제는 정말 중요한 지적이에요. 이건 제가 추가 제안을 드릴게요: **textColorOverride 적용 범위를 제한하자.** `onSurface`와 `onSurfaceVariant`에만 적용하고, `onError`, `onPrimary`, `onSecondary` 등 시맨틱 슬롯은 엔진이 자동 계산하게 남겨두자는 거예요. 유저가 원하는 건 '카드 위의 문구 색'이지 '에러 메시지 색'이 아니에요."

이러면 `onPrimary`, `onError` 등은 HCT 엔진이 생성한 원래 값을 유지해요. **하지만** 유저 관점에서 '글자색을 골드로 바꿨는데, 버튼 위 텍스트는 여전히 흰색이네?' 하는 일관성 깨짐이 생겨요. 팬은 '내 앱 전체가 이 글자색'을 기대할 수 있거든요.

절충안: **`onSurface`, `onSurfaceVariant`, `onPrimaryContainer`, `onSecondaryContainer`, `onTertiaryContainer`에만 적용하고, `onPrimary`, `onSecondary`, `onTertiary`, `onError`, `onErrorContainer`는 건드리지 않기.** 이유: container 위 텍스트는 주로 카드, 칩 내부의 문구라 '최애색 글자'가 자연스러운 맥락이고, Primary/Secondary 버튼 위 텍스트는 시스템 액션(저장, 취소) 맥락이라 시맨틱 유지가 중요하니까요."

**소영:**

"준서 님 절충안 — 합리적이에요. Container 계열은 콘텐츠 영역이고 Primary/Error는 액션 영역이니까, 분리하는 게 Material 시맨틱에 부합해요.

hex-HCT 동기화 문제는... 사실 **양방향 동기화를 포기하면** 간단해져요. hex 입력은 hex 입력대로 색을 결정하고, 슬라이더는 슬라이더대로. 둘이 동시에 보이되 **마지막으로 조작한 쪽이 우선(last-write-wins)**하면 돼요. hex 입력하면 슬라이더 위치가 업데이트되지만, 그 역은 하지 않아요. 슬라이더 움직이면 hex 필드가 실시간으로 업데이트되고. **hex -> 슬라이더 방향만 동기화, 슬라이더 -> hex 방향도 동기화. 양방향이지만 마지막 입력 우선.** HCT 클램핑으로 미세 차이가 생겨도, hex 필드에 '실제 적용될 색'의 hex를 보여주면 유저가 혼란 없이 인지해요."

## 핵심 인사이트

> **hex 입력 = 팬의 '정확한 색 재현' 욕구 해소.** K-pop 팬은 공식 컬러 hex를 외우고 다닌다. 슬라이더로 근사값을 찾는 것과 hex 6자리를 붙여넣는 건 체감 자유도가 완전히 다르다.

> **2D 피커는 배경색에서만, 글자색은 hex+슬라이더.** 글자색의 핵심은 '가독성'이지 '탐색'이 아니다. hex로 정확한 값을 입력하거나, WCAG 추천에서 고르거나, 슬라이더로 미세 조정하는 3가지 경로면 충분하다.

> **textColorOverride = 콘텐츠 슬롯만.** 에러/액션 버튼의 글자색은 시맨틱 무결성을 위해 엔진 자동에 맡긴다. 유저가 원하는 '내 글자색'은 카드, 문구, 본문 텍스트에 적용되면 충분하다.

> **WCAG 경고 = 차단 없이 시각적 피드백 극대화.** < 3.0 프리뷰 오버레이는 '확인 다이얼로그'의 마찰 없이 유저 스스로 문제를 인지하게 한다. 팬 정체성 인프라의 자율성 원칙과 접근성 원칙의 교차점.
