## 토론 요약

### Topic 1: A+B 이중 방어 vs 단일 방어

**A조 (도현, 준서) — 이중 방어 필수:**
- `saveToSlot()`에 IAP 검증 없음 — 모든 UI 진입점 차단하는 단일 보안 경계 필요
- SharedPrefs 평문 저장이라 저장 1회 우회 = 영구 무료
- Google Play billing bypass 분류 → 앱 정지 사유 가능
- 환불 시나리오: 구매→저장→환불 시 커스텀 테마 그대로 남음
- `revokeThemePicker()` 미구현 상태

**B조 (서아, 하윤) — B는 과잉, 업셀 패턴으로:**
- A만으로 데이터 무결성 100% → B(버튼 비활성화)는 수익 차단
- 프리뷰 중 저장 시도 = 구매 의향 시그널 → 비활성화하면 전환 기회 0
- 팬의 "내가 만든 것 = 내 것" 심리 → 비활성 버튼은 좌절, 구매 시트는 완성
- 10대 유저 "닫기" 반사 0.5초 vs 업셀 시트 3~5초

**재반론 → 수렴 (도현):**
- `setCustomColor()`가 즉시 `_persistCurrentState()` 호출 → 프리뷰 중 앱 킬 시 커스텀 색상 영구 잔류
- 따라서 A만으로 부족 — persist 레벨 방어 필수
- **B를 "버튼 비활성화"에서 "persist 게이트"로 재정의** → 전원 동의

### Topic 2: 프리뷰 → IAP 전환 UX

- 프리뷰 무제한 유지 (제한하면 이탈)
- 시트 닫기 시 감성 업셀 바텀시트 1회 표시
- 카피 방향: "방금 만든 이 색상, 저장할 수 있어요"
- `barrierDismissible: true` (강제감 없이)
- 프리뷰 모드 상단 Chip 라벨 (Dark Pattern 가이드라인 방어)

### Topic 3: 환불 후 데이터 무결성

- 슬롯 데이터 삭제 반대 → 소프트 락 (데이터 유지 + 기능 차단)
- `applySlot()`에서 custom+미구매 → 기본 팔레트 폴백
- 재구매 시 즉시 복원 = 재구매 훅
- `revokeThemePicker()` P2로 구현
- 서버 RTDN은 Phase 7 구독 도입 시 함께
