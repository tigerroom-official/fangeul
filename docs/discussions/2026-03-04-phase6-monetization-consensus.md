## 2. 확정 합의 사항

### BM 구조

1. **버블 기본 기능(변환, 키보드, 띄우기)은 영구 무료.** 핵심 엣지를 페이월 뒤에 두는 순간 입소문/바이럴이 차단된다. 전원 재확인.

2. **MVP 수익 구조 = 광고(배너+보상형) + IAP 단일.** 구독은 Phase 7 이후. MVP에서 구독 인프라는 코드 레벨로만 준비.

3. **IAP 상품 구조 = 감성 컬러 팩 (단독) + 언어별 문구 번들 (병행).** 단일 SKU 대신 두 축 병행이 전환율 우위:
   - 감성 컬러 팩: 버블 스킨 + 문구 50개 + 발음 30개, ₩1,900 단독 / ₩3,900 2팩
   - 언어별 문구 번들: 특정 언어권 팬 집중 소비 패턴 대응 (컴백 시즌 한정)
   - 명명 원칙: 색상+감성 조합 ("퍼플 드림", "골든 아워", "그날 콘서트 하늘", "새벽 응원봉 잔광" 등)
   - IP 안전: 아이돌명·팬덤명·로고·이미지 일체 미사용, "~팬을 위한" 문구도 금지

4. **퍼널 단계적 노출 원칙.** 유저에게 노출하는 순서가 단계적이라는 의미. Phase 6 구현 범위에는 광고+IAP+퍼널이 모두 포함됨:
   - 유저 노출 1단계 (Day 7~): 보상형 광고만 노출
   - 유저 노출 2단계 (Day 14~): IAP 트리거 추가 노출
   - 유저 노출 3단계 (Day 21+): 배너 조건부 진입
   - ※ "MVP=IAP 단일"은 **구독 제외**라는 뜻. 광고+IAP는 Phase 6 범위에 포함

5. **IAP 전용 콘텐츠 일부는 반드시 보상형으로 해금 불가하게 설정.** 희소성이 무너지면 IAP 전환율도 무너진다.

### 유저 가치사다리

6. **UVP = "한국어 도구"가 아니라 "팬 정체성 인프라".** 포지셔닝 전환:
   - Before: "한글 타이핑/변환 앱"
   - After: "내 최애와 연결되는 언어 공간"
   - 결제 근거: 기능 사용료가 아닌 팬 정체성 표현 비용

7. **무료 유저 경험 제한 구조:**
   - 즐겨찾기: **허니문(Day 0~6) 무제한, Day 7부터 3슬롯** (Pro = 무제한)
   - TTS: 5회/일 (Pro = 무제한 + 느린 발음 + 버블 내 TTS)
   - 문구 팩: 기본팩 무료 접근, 감성 컬러 팩은 보상형 4h 해금 or IAP
   - 멤버 이름 치환(`{{member_name}}`): 영구 무료 — 개인화 기능은 입소문 엔진
   - **저장소 규칙:** 즐겨찾기 슬롯 카운트, TTS 일일 카운터, 허니문 플래그, 해금 타이머 → 모두 `flutter_secure_storage` + HMAC 서명 (rules §보안 준수)

8. **수익 구조 비율 목표 (안정화 후): 광고 70% + IAP 30%.** 초기 광고 편중(85:15)에서 IAP 비중 단계적 확대.

### 동남아 수익 전략

9. **세그먼트별 차별 전략:**
   - 무결제(70%): 보상형 광고 — 광고 시청 그 자체가 수익
   - 저관여(15~20%): 초저가 IAP (₩990 진입 SKU) — "포카 한 장 값" 앵커링
   - 고관여(5~10%): 프리미엄 팩 풀세트 구매 유도
   - 비활성(~5%): 설치 후 이탈 — 수익 기여 없음

10. **Google Play 잔액 경로 활용.** GoPay/OVO → Google Play 잔액 충전 경로가 동남아 유일한 현실적 결제 수단. Play Store 현지 가격 필수:
    - ID: IDR 12,000 (₩990 상당)
    - PH: PHP 39
    - BR: BRL 5.90
    - 영미권: $1.99~2.99

### 전환 퍼널

11. **허니문 → 보상형 습관 → IAP 트리거 3단계 퍼널 (변경 확정):**

    | 구간 | 기간 | 전략 |
    |------|------|------|
    | 허니문 | Day 0~6 | 광고 없음. 전체 기능 자유 체험. 최애 선택 + 개인화 완성 유도 |
    | 보상형 습관 | Day 7~13 | 보상형 광고 3회/일 시작. "팬 패스 획득" 보상 프레이밍. 제한 상기 없음 |
    | IAP 트리거 | Day 14~21 | 보상형 3회 소진 + 즐겨찾기 슬롯 포화 동시 달성 시 단일 IAP 팝업 |
    | 이후 | Day 22+ | 컴백/생일 시즌 이벤트 + 스트릭 마일스톤 할인 쿠폰 |

12. **D-day 선물 정책.** 유저가 등록한 최애 생일 / 컴백 날 = 당일 모든 제한 24h 전면 해제. 선물로 인식되면 팬 감성 강화 + 자발 공유 유도. 구현 비용 대비 감성 ROI 최고.

13. **MVP 퍼널 = 3개만 (슬롯 제한 + 보상형 + IAP).** 복잡도 최소화 원칙. 미션 기반 영구 해금, 스트릭 할인 등은 v1.1+ 단계.

### 보상형 광고 설계 (2/28 유지 항목)

14. **해금 시간 = 4시간 (or 자정 중 빠른 것).** 3회/일 상한. "팬 패스 획득!" 보상 프레이밍. 변경 없음.

15. **배너 조건부 유지 → 단계적 제거.** 보상형 1회 시청 시 세션 배너 제거. Pro/IAP 구매 시 영구 제거. DAU 15% 전환 시 전면 종료.

## 6. DAU별 수익 예측 테이블

### 가정 (교차 리뷰 반영 수정)
- 광고: 배너 eCPM $0.3 (동남아), 보상형 eCPM $3.0 (동남아), fill rate 80%
- 배너 노출: DAU × 2회/일 × fill rate 80%
- 보상형 시청률: DAU의 8% (업계 평균 6~10%)
- **IAP: 일회성 구매이므로 월 신규 설치 기반으로 계산** (기존 DAU 일간 전환 계산 오류 수정)
- IAP 전환율: **월 신규 설치의 2~5%** (동남아 보수적 2%, 낙관 5%)
- IAP ARPU: $1.20 (₩1,900 기준, Play Store 수수료 30% 제외)
- 월 신규 설치 = DAU × 2~3배 추정 (초기 성장기), 안정기에는 DAU × 0.5배
- 지역 가중: 동남아 70% / 기타 30%

## 9. 운영 시나리오 명세 (교차 리뷰 지적 보완)

### 9.1 IAP 결제

| 시나리오 | 처리 |
|---------|------|
| **결제 성공** | `PurchaseDetails.status == purchased` → 콘텐츠 해금 + secure_storage HMAC 저장 + `completePurchase()` |
| **결제 실패** | 에러 메시지 표시 + 재시도 버튼. 3회 실패 시 "나중에 다시 시도" 안내 |
| **결제 Pending** (캐리어 빌링, 느린 네트워크) | "처리 중" UI + 다음 앱 진입 시 `restorePurchases()` 자동 호출로 상태 확인 |
| **환불 발생** | Google Play RTDN(Real-Time Developer Notification) 수신 시 콘텐츠 잠금. 서버 없으면 다음 앱 진입 시 `restorePurchases()` → 구매 목록에 없으면 잠금 |
| **기기 변경/재설치** | 설정 화면 "구매 복원" 버튼 → `restorePurchases()` → 복원 성공 시 secure_storage 재생성 + HMAC |
| **오프라인 구매 시도** | "인터넷 연결 필요" 안내 + 보상형 광고 대안 제안 |

### 9.2 보상형 광고

| 시나리오 | 처리 |
|---------|------|
| **AdMob SDK 초기화 실패** | 30초 후 자동 재시도 (최대 3회). 실패 시 "팬 패스" 버튼 숨김, 배너 영역 빈 공간 |
| **광고 로드 실패/fill rate 0** | "지금은 광고가 없어요. 잠시 후 다시 시도해주세요" + 30분 후 재시도 |
| **광고 시청 중 앱 킬** | `onUserEarnedReward` 미호출 → 보상 미지급 (정상 동작) |
| **보상형 타이머 중 앱 킬** | 만료 타임스탬프가 secure_storage에 저장됨 → 재진입 시 현재시간과 비교, 미만료면 해금 유지 |
| **오프라인 상태** | 광고 로드 불가 → "팬 패스" 비활성 + "인터넷 연결 시 사용 가능" 안내 |

### 9.3 D-day 선물 정책

| 시나리오 | 처리 |
|---------|------|
| **정상 해금** | 캘린더 기반 최애 이벤트 당일 → 24h 해금 + secure_storage에 `{date}_{eventId}` 기록 |
| **기기 시간 조작** | 단조증가 검증: `currentTimestamp < lastRecordedTimestamp` → 해금 거부 |
| **같은 이벤트 중복 해금** | secure_storage에 해금 이력 조회 → 이미 기록 있으면 거부 |
| **여러 이벤트 같은 날** | 해금은 1회만 (가장 먼저 트리거된 이벤트) |
