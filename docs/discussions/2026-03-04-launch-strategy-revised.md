## Topic R1: 글로벌 동시 출시 vs KR 우선 유지

### A조 선공

**민지 (Growth PM):**

> "시그널을 읽으세요."

- ytaro US 16%, SEA+JP 8%가 마케팅 0 오가닉 유입 — 풀(pull) 수요 존재
- Fangeul의 코어 유저는 구조적으로 비한국어권 K-pop 팬 — KR 우선은 타겟 미스매치
- AI 구독 $400/월로 번역+스크린샷 해결 — 300~500만원 번역비/100~200만원 외주비 소멸
- KR만 열면 이미 작동 중인 오가닉 채널을 인위적으로 차단하는 것

**수아 (팬덤 커뮤니티):**

> "팬은 '내 아이돌을 위한 도구'면 기꺼이 퍼뜨립니다."

- 팬덤 바이럴은 글로벌 커뮤니티(X, 인스타, 틱톡, 레딧) → KR 방향
- 인도네시아/태국/필리핀 팬이 언어 장벽 때문에 더 열심히 한국어 공부 — Fangeul의 정확한 타겟
- 팬덤은 이벤트 중심(생일, 컴백, 콘서트) — 골든 모멘트에 앱 접근 불가하면 기회 상실

### B조 반론

**도윤 (회의론자):**

> "36명 데이터로 전략을 세우면 확증편향에 빠집니다."

- US 16% = 5.76명, 95% 신뢰구간 [4%, 28%] — 통계적으로 노이즈
- AI 번역 품질: 태국어 팬덤 은어, 인도네시아 K-pop 용어는 AI가 못 잡음
- soft launch 구조(국가별 단계적 롤아웃)가 맞음

**하준 (인디앱 개발자):**

> "Play Store에 올리고 2주 보세요."

- Play Store 지역 설정은 비가역적 결정이 아님 — 언제든 변경 가능
- 현실적 Phase 7 작업량: 언어당 검수 루프 2~4시간, 5개 언어 = 10~20시간
- 제안: KR + EN-US 2개로 시작, 2주 후 SEA 추가

### A조 재반론

**민지:**
- "전 세계 200개국 Day 1"이 아니라 국가 제외를 안 하는 것 — Play Store 기본값이 글로벌
- 스토어 설명 영어 하나만 잘 쓰면 SEA 팬도 찾아옴

**수아:**
- Day 1: KR + EN (스토어 설명 2개만)
- Day 14: 인도네시아어 + 태국어 추가
- Day 30: 필리핀 + 일본어

### 사회자 정리

**합의:** KR 단독 부적절 (전원 동의). 국가 제외 안 함 = 사실상 글로벌 기본.
**결론:** Day 1 KR+EN 스토어 설명 + 국가 제외 없음 → Week 2 ID/TH 추가 → Week 4 데이터 보고 결정.

## Topic R2: 즐겨찾기 슬롯 정책 최종 결정

### A조 선공 (느슨한 정책)

**수아:**
- 팬은 최소 5번 저장해야 "쓸 만하다" 각인 — 3슬롯은 첫 분류에서 막힘
- Day 7은 팬덤 사용 패턴(컴백 2~4주 주기)과 불일치
- Day 7에 벽 → 그냥 삭제

**하준:**
- DAU 10~50명에서 수익화 퍼널 검증 불가능 — 샘플 부족
- 이 단계 목표: "바이럴 가능한 제품인가" 확인
- 코드 변경 2줄, 개발 비용 제로

### B조 반론 (현행 유지)

**재혁:**
- 목표는 "돈을 내는 유저"를 잡는 것
- 느슨한 정책의 함정: 기대치 높게 설정 → 나중에 강화 시 이탈
- Day 7 리텐션은 모바일 앱 산업 표준 (20~25%)

**도윤:**
- A안(3슬롯/Day 7)이 "더 빠르게 틀렸다는 걸 안다"
- 데이터 없는 상태에서 최적화 불가 — 가설 세우고 검증하는 것만 가능
- 단, DAU 10~50에서 슬롯 효과를 통계적으로 분리할 수 없음

### A조 재반론

**수아:** 3슬롯에 막히면 자랑할 경험이 끊김 → 바이럴 차단
**하준:** 5슬롯/Day 21은 앱 매력도를 더 깨끗하게 측정 가능

### 사회자 최종 권고 (민지)

**리스크 비대칭성 분석:**
- B안(느슨) 전환율 낮으면 → 슬롯 조이면 됨 (코드 2줄)
- A안(타이트) 유저 없으면 → 되돌릴 방법 없음

**최종 결론: 수정된 B안**

```
즐겨찾기 슬롯: 5개
허니문 기간: Day 14 (14일간 무제한)
슬롯 제한 시작: Day 15~
조건부 재검토: Day 30 광고 시청률 < 20% → 슬롯 4개로 조정
```

## Topic R3: AI 레버리지 극대화

### A조 (적극 활용)

**민지:**
1. 자체 분석 대시보드 P2→P0: Firebase → VPS ETL, Grafana — 3차원 코호트 교차 분석
2. AI 프로모 에셋: ComfyUI로 앱 스크린샷/배너 자체 생산 (아이돌 콘텐츠 아님)
3. 멀티 리전 ASO: Claude + Codex 교차검증

**재혁:**
1. 자체 수익 대시보드: 일별 IAP 전환율, 코호트별 광고 소진율, Day 7/14 퍼널 드롭
2. ytaro 동적 카드: 정적→행동 트리거 기반 (전환율 차이)
3. Firebase Remote Config: 슬롯 메시지 등 변수화 — 앱 업데이트 없이 변경

### B조 (과잉 경고)

**도윤:**
- DAU 0에서 Grafana = 빈 그래프
- "이틀"이 일주일 되고 출시 2주 지연될 수 있음
- 역량이 있다 ≠ 지금 써야 한다

**수아:**
- 팬은 AI 생성 콘텐츠를 귀신같이 감지 — 커뮤니티 신뢰 리스크
- ASO는 번역 품질이 전부가 아님 — 현지 팬덤 용어는 AI가 모름
- 팬 커뮤니티 신뢰는 속도가 아닌 진정성

### A조 재반론

**민지:** 출시 전 완성이 아니라 파이프라인 준비. 데이터 들어오기 시작하면 바로 볼 수 있는 상태. AI 에셋은 UI 비주얼만 (팬 아트/아이돌 콘텐츠 절대 금지).

**재혁:** Remote Config = A/B 테스트 인프라가 아니라 변경 유연성. 앱 업데이트 없이 프레이밍 변경 가능. ytaro 동적 카드는 크로스앱 내부 로직 수정 — 외부 의존성 없음.

### 합의

**추가 Action Items:**
- P0: Firebase Remote Config 변수화
- P1: 수익 모니터링 대시보드 (VPS+PostgreSQL+Grafana, 출시와 동시 가동)
- P1: Play Store 에셋 AI 생산 (UI 비주얼만)
- P1: ytaro 동적 카드 (행동 트리거)
- P2: A/B 테스트 준비 (DAU 500+ 시 가동)

**Not-To-Do:**
- 팬/아이돌 AI 생성 콘텐츠 (팬 커뮤니티 신뢰 리스크)
- DAU 500 전 A/B 테스트 실행
- ytaro 대규모 리아키텍처
